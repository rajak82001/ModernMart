One architectural decision I had to make early was whether to keep the cart state simple with local component state or move it into a shared context with a reducer. I considered both options. Component state would have been quicker to wire up for a small assignment, but it would have made the cart drawer, product cards, and product detail page harder to keep in sync as the app grew. A context + reducer setup added a bit more structure upfront, but it gave the app a more predictable state flow and made the cart behavior easier to reason about across multiple surfaces. I chose the shared state approach because the assignment already required a cart that could be opened from different places and updated consistently.

I also had to decide how to handle product variants. The fake API only returns one image and a basic product payload, so I could either keep the experience minimal or supplement it with local variant metadata. I chose the latter because it made the detail page feel much closer to a real storefront and gave the UI something meaningful to interact with. That said, it also introduced a tradeoff: the data is more realistic, but it is still mock-driven rather than backed by a real inventory system.

With more time, I would clean up a few things. I would add a proper backend or API layer for cart and inventory persistence, improve test coverage for the key user flows, and make the product data model more robust so variant logic is less dependent on local fallback data. I would also spend more time refining the styling system and accessibility details so the app feels more polished and production-ready.
